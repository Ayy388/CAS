### 1.1 Redis 选型理由

| 场景 | 不使用 Redis 的问题 | 使用 Redis 后的改善 |
|------|-------------------|-------------------|
| 课程大厅列表（高频读取） | 每次页面加载/刷新都查询 MySQL，30 门课 x 多个 JOIN | 缓存 5 分钟，QPS 从 1000↓ 到接近 0 DB 查询 |
| 当前活跃活动（每 30s 轮询） | 每个学生每 30s 查一次 MySQL | 缓存 30s，所有学生走缓存 |
| 抢课计数器（高并发写入） | 每次 UPDATE 都写 InnoDB 行锁 | Redis INCR/DECR 纯内存操作，异步落库 |
| Token 黑名单 | 无法主动让 Token 失效 | Redis SET 存储黑名单，TTL 自动过期 |

### 1.2 RabbitMQ 选型理由

| 场景 | 不同步处理 | 使用 MQ 后 |
|------|-----------|-----------|
| 开课审核 → 批量通知（可能 1000+ 条） | 管理员等待 INSERT 完成 | 异步处理，页面立即返回"审核成功" |
| 抢课记录 | 抢课流程中不涉及 | 抢课成功后发送消息，消费者异步完成后续操作（如统计更新、日志记录） |
| 活动状态变更 | 定时任务轮询（1 分钟延迟） | 也可通过时间消息延迟队列实现更精确的 PENDING→ACTIVE 切换 |

## 二、Redis 详细设计

### 2.1 缓存架构

```
┌─────────────────────┐
│     Vue 前端         │
│  每 30s 轮询活动状态  │
│  每次进入课程大厅刷新  │
└────────┬────────────┘
         │ HTTP
         ▼
┌────────────────────────────────────────┐
│          Spring Boot 后端               │
│                                         │
│  ┌──────────────────────────────┐      │
│  │     Redis 缓存层              │      │
│  │                              │      │
│  │  Cache-Aside 模式：           │      │
│  │  读：先查 Redis → 命中返回    │      │
│  │      → 未命中查 DB → 写 Redis │      │
│  │  写：更新 DB → 删除 Redis     │      │
│  └──────────────────────────────┘      │
│                                         │
│  ┌──────────────────────────────┐      │
│  │     MySQL 数据库              │      │
│  └──────────────────────────────┘      │
└────────────────────────────────────────┘
```

### 2.2 缓存键设计

| 缓存键 | 类型 | 存储内容 | TTL | 说明 |
|--------|------|---------|-----|------|
| `cas:course:list:{type}:{page}` | String(JSON) | 课程大厅分页列表 | 5min | 按类型+页码缓存 |
| `cas:campaign:current` | String(JSON) | 当前活跃活动完整信息 | 30s | 每 30s 刷新 |
| `cas:offering:{id}` | String(JSON) | 单个课程详情（含实时名额） | 30s | 课程详情页 |
| `cas:enrollment:count:{offeringId}` | String | 课程已报名人数（纯数字） | 10s | 抢课按钮 实时刷新 |
| `cas:token:blacklist:{jti}` | String | "1" | Token 剩余 TTL | 登出/踢下线 |
| `cas:enroll:request:{studentId}` | String | timestamp | 2s | **抢课去重：同一学生 2s 内只能抢一次** |

### 2.3 缓存更新策略

| 操作 | 缓存动作 | 说明 |
|------|---------|------|
| 管理员新增/编辑/删除课程 | `DEL cas:course:list:*` | 删除所有课程大厅缓存 |
| 管理员创建/编辑开课 | `DEL cas:course:list:*` + `DEL cas:offering:{id}` | 课程列表 + 课程详情 |
| 学生抢课成功 | `DEL cas:course:list:*` + `DEL cas:offering:{id}` + `SET cas:enrollment:count:{offeringId}` | 名额变了，更新实时计数 |
| 学生退课 | `DEL cas:course:list:*` + `DEL cas:offering:{id}` + `DECR cas:enrollment:count:{offeringId}` | 名额变了 |
| 管理员审核通过/驳回 | `DEL cas:course:list:*` + `DEL cas:offering:{id}` | 状态变了 |
| 活动状态变更 | `DEL cas:campaign:current` + `DEL cas:course:list:*` | 活动+课程列表状态变了 |

### 2.4 抢课计数器（Redis 优化方案）

在原有的原子 UPDATE 方案之上，增加 Redis 计数器作为前置过滤：

```
请求抢课
  │
  ▼
① Redis 请求去重：SET cas:enroll:request:{studentId} 1 NX EX 2
   → 已有值 → 429 "操作太频繁"
  │
  ▼
② Redis 容量预检：GET cas:enrollment:count:{offeringId}
   → >= maxCapacity → 409 "已满员"（快速短路，不查 DB）
  │
  ▼
③ Redis INCR 临时计数：INCR cas:enrollment:temp:{offeringId}
   → 返回值 > maxCapacity → REDIS DECR + 409 "已满员"
  │
  ▼
④ MySQL 原子 UPDATE（兜底保证）
   UPDATE course_offering
      SET enrolled_count = enrolled_count + 1
    WHERE id = ? AND enrolled_count < max_capacity
   → affectedRows = 0 → REDIS DECR → 409
  │
  ▼
⑤ INSERT enrollment + 删除缓存
  │
  ▼
⑥ DECR 临时计数：DECR cas:enrollment:temp:{offeringId}
  │
  ▼
⑦ MQ 发送抢课成功消息（异步记录日志、更新统计）
  │
  ▼
返回成功
```

> **关键设计说明**：INCR 临时计数器代表"当前正在处理中的请求数"，不是"已成功抢课总数"。**无论请求最终成功还是失败，都必须 DECR 释放槽位**，否则临时计数器会永久膨胀，导致后续请求全部被误杀。
>
> **为什么 MySQL 事务还是必需的**：Redis 是最终一致性，可能因宕机丢失 INCR 数据。MySQL 原子 UPDATE 作为最终校验保证"绝不超卖"。Redis 计数器减少了 80% 不必要的 MySQL UPDATE 冲突。

## 三、RabbitMQ 详细设计

### 3.1 消息架构

```
┌──────────────────┐     ┌───────────────┐     ┌──────────────────┐
│  生产者 (Producer) │     │  RabbitMQ     │     │  消费者 (Consumer)│
│                   │     │               │     │                  │
│  EnrollmentService│────▶│  exchange     │────▶│  Notification    │
│  ReviewService    │     │  + queue      │     │  生成通知        │
│  CampaignScheduler│     │               │     │  更新统计        │
└──────────────────┘     └───────────────┘     │  日志记录        │
                                               └──────────────────┘
```

### 3.2 交换机与队列定义

| 交换机名称 | 类型 | 绑定队列 | 路由键 | 说明 |
|-----------|------|---------|-------|------|
| `cas.exchange.direct` | Direct | `cas.queue.notification` | `notification` | 审核后异步发通知 |
| `cas.exchange.direct` | Direct | `cas.queue.enrollment.log` | `enrollment.log` | 抢课日志记录 |
| `cas.exchange.delayed` | Delayed | `cas.queue.campaign.start` | `campaign.start` | 活动开始时自动 PENDING→ACTIVE |

### 3.3 消息流程

**场景一：开课审核 → 异步通知**

```
ReviewService.approve(offeringId)
  │
  ├── ① 更新 offering.status + enrollment.status（同步，事务内）
  │
  ├── ② 发送 MQ 消息（事务提交后发送）
  │      exchange: cas.exchange.direct
  │      routingKey: notification
  │      payload: { offeringId, type: "APPROVED" }
  │
  └── ③ 返回前端 "审核成功，通知正在发送"
         │
         ▼  (异步)
  NotificationConsumer
  │
  ├── 接收消息
  ├── INSERT INTO notification ... SELECT ...（批量 SQL）
  └── 完成
```

**场景二：抢课成功 → 异步日志**

```
EnrollmentService.enroll()
  │
  ├── ① 事务内完成抢课
  │
  ├── ② 事务提交后发送 MQ 消息
  │      routingKey: enrollment.log
  │      payload: { enrollmentId, studentId, offeringId, timestamp }
  │
  └── ③ 返回前端 "抢课成功"
         │
         ▼  (异步)
  EnrollmentLogConsumer
  │
  ├── 更新 Redis 统计缓存
  └── 写入操作日志表（V2.0 预留）
```

### 3.4 事务消息一致性

```java
// 使用 Spring @TransactionalEventListener 确保事务提交后再发消息
@Service
public class EnrollmentService {

    @Transactional
    public EnrollResponse enroll(...) {
        // ... 抢课业务逻辑
        // 事务内不发送消息
        return response;
    }

    // 事务提交后自动触发
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onEnrollSuccess(EnrollEvent event) {
        // ① DECR 临时计数，释放槽位（无论成功/失败都要释放）
        redisTemplate.opsForValue().decrement("cas:enrollment:temp:" + event.getOfferingId());
        // ② 发送 MQ 消息
        rabbitTemplate.convertAndSend("cas.exchange.direct",
            "enrollment.log", event);
    }

    // 事务回滚后处理
    @TransactionalEventListener(phase = TransactionPhase.AFTER_ROLLBACK)
    public void onEnrollFailed(EnrollEvent event) {
        // Redis 计数器回滚：DECR 临时计数
        redisTemplate.opsForValue().decrement("cas:enrollment:temp:" + event.getOfferingId());
    }
}
```

### 3.5 避免消息重复消费

| 措施 | 实现 |
|------|------|
| 消费者幂等 | 通知表无唯一约束时，消费前先查询是否已生成 |
| 手动 ACK | `channel.basicAck()`，处理失败重回队列或死信队列 |
| 消息去重 | 生产者每个消息携带唯一 `messageId`，消费者记录已处理 ID |

### 3.6 RabbitMQ 配置


| **campaign** | admin+公共 | 活动 CRUD、开启/结束 | 审核（review） |

### 5.2 模块间协作

```
enrollment.enroll() 完整流程涉及的模块和中间件：

EnrollmentController
  → EnrollmentService
      → CampaignService（查询当前活跃活动）
      → Redis（请求去重 + 计数器预检）
      → OfferingService（原子 UPDATE 容量）
      → UserService（校验开放范围）
      → MySQL INSERT enrollment
      → Redis（删除相关缓存，DECR 临时计数）
      → RabbitMQ（发送 enrollment.log 消息）
      → 返回响应

review.approve() 完整流程：

ReviewController
  → ReviewService
      → OfferingService（更新状态）
      → EnrollmentService（批量更新 enrollment）
      → Redis（删除缓存）
      → RabbitMQ（发送 notification 消息）
      → 返回响应
      → (异步) NotificationConsumer → MySQL INSERT 通知
```

## 六、完整 API 设计

### 6.1 统一约定

| 项目 | 规范 |
|------|------|
| 基础路径 | `/api/v1` |
| 请求体 | JSON, `Content-Type: application/json` |
| 响应格式 | 统一 `ApiResponse<T>` 包裹 |
| 认证方式 | `Authorization: Bearer <token>` |
| 分页参数 | `?page=1&pageSize=20` |
| 日期格式 | `yyyy-MM-dd HH:mm:ss` |
| 字段命名 | 请求/响应 camelCase（匹配前端 TypeScript 类型） |

### 6.2 响应体与前端类型对齐

```typescript
// 前端 TypeScript → 后端 JSON 响应必须一致

// ApiResponse<T>  →  { code: 200, message: "success", data: ... }
// User            →  { id, username, realName, role, department, major, grade }
// Semester        →  { id, name, academicYear, semesterType, startDate, endDate, status }
// Course          →  { id, code, name, type, credits, hours, description, createdAt }
// CourseOffering  →  { id, semesterId, courseId, courseName, courseType, teacherId,
//                      teacherName, maxCapacity, minEnrollment, seatsRemaining,
//                      openGrade, openMajor, location, schedule, status }
// SelectionCampaign → { id, name, semesterId, semesterName, startTime, endTime, status }
// Enrollment      →  { id, campaignId, offeringId, offeringName, studentId,
//                      studentName, status, enrolledAt, reviewedAt? }
// Notification    →  { id, type, title, content, courseName, createdAt, read }
```

**字段对齐关键点**：
- 前端 `seatsRemaining` = 后端 VO 计算 `maxCapacity - enrolledCount`
- 前端 `read` = 数据库 `is_read`（VO 层转换）
- 前端 `courseName` / `teacherName` / `offeringName` / `semesterName` = VO 层 JOIN 组装
- 所有枚举字符串值必须与前端字面量一致

## 七、核心业务流程完整实现

### 7.1 抢课（含 Redis + MQ）

```
POST /api/v1/enrollments
       │
       ▼
EnrollmentController.enroll(@RequestBody EnrollRequest req)
  │ 获取当前用户 studentId（从 JWT Token）
  │
  ▼
EnrollmentServiceImpl.enroll(offeringId, studentId)
  │
  ├── [Redis] ① 请求去重
  │      SET cas:enroll:request:{studentId} 1 NX EX 2
  │      → 已存在 → 429 "操作太频繁，请稍后再试"
  │
  ├── [DB] ② 查询当前活跃活动
  │      campaignService.getCurrentActive()
  │      → 不存在 → 409 "当前没有进行中的选课活动"
  │      → status != ACTIVE → 409 "选课活动未开始或已结束"
  │
  ├── [DB] ③ BR-03 校验：同活动下是否已选课
  │      → 已存在 ENROLLED/APPROVED → 409 "您已选课"
  │
  ├── [DB] ④ BR-05 校验：开放范围
  │      → 不匹配 → 409 "您不符合开放范围"
  │
  ├── [Redis] ⑤ 容量预检（快速短路）
  │      GET cas:enrollment:count:{offeringId}
  │      → >= maxCapacity → 409 "已满员"（LDECR）
  │
  ├── [Redis] ⑥ INCR 临时计数（分布式环境下防止并发超卖）
  │      INCR cas:enrollment:temp:{offeringId}
  │      → 超过 maxCapacity → DECR + 409 "已满员"
  │
  ├── [MySQL] ⑦ 原子 UPDATE（最终一致性保证）
  │      UPDATE course_offering
  │         SET enrolled_count = enrolled_count + 1
  │       WHERE id = ? AND enrolled_count < max_capacity
  │      → affectedRows = 0 → Redis DECR → 409 "已满员"
  │
  ├── [MySQL] ⑧ INSERT enrollment
  │      INSERT INTO enrollment (...) VALUES (...)
  │
  ├── [Redis] ⑨ DECR 临时计数（释放 INCR 槽位）
  │      DECR cas:enrollment:temp:{offeringId}
  │
  ├── [Redis] ⑩ 删除相关缓存
  │      DEL cas:course:list:*  +  DEL cas:offering:{id}
  │      SET cas:enrollment:count:{offeringId} 新数值
  │
  ├── 事务提交
  │
  ├── [MQ] ⑪ 事务提交后发送事件
  │      rabbitTemplate.convertAndSend("cas.exchange.direct",
  │          "enrollment.log", new EnrollEvent(enrollmentId))
  │
  └── 返回 200 { enrollmentId, message: "抢课成功" }
```

### 7.2 退课

```
DELETE /api/v1/enrollments/{id}
       │
       ▼
EnrollmentServiceImpl.drop(enrollmentId, studentId)
  │
  ├── [DB] ① 查询 enrollment，校验归属
  │      → studentId 不匹配 → 403
  │
  ├── [DB] ② 校验退课时间
  │      → campaign.status = 'ENDED' → 409 "已超过退课时间"
  │
  ├── [MySQL] ③ 原子 DECREMENT
  │      UPDATE course_offering
  │         SET enrolled_count = enrolled_count - 1
  │       WHERE id = ? AND enrolled_count > 0
  │
  ├── [MySQL] ④ DELETE enrollment
  │
  └── [Redis] ⑤ 删除相关缓存
         DEL cas:course:list:* + DEL cas:offering:{id}
         DECR cas:enrollment:count:{offeringId}
```

### 7.3 开课审核（含 MQ 异步通知）

```
POST /api/v1/admin/review/{id}/approve
       │
       ▼
ReviewServiceImpl.approve(offeringId)
  │
  ├── [MySQL] ① 更新 offering.status = 'APPROVED'
  │
  ├── [MySQL] ② 批量更新 enrollment.status = 'APPROVED'
  │
  ├── [Redis] ③ 删除缓存
  │      DEL cas:course:list:*
  │
  ├── 事务提交
  │
  └── [MQ] ④ 事务提交后发送通知消息
         rabbitTemplate.convertAndSend("cas.exchange.direct",
             "notification",
             new ReviewEvent(offeringId, "APPROVED"))

         ── 异步 ──▶ NotificationConsumer
                       → INSERT INTO notification SELECT ...（按 offering_id 批量 SQL）
                       → 完成

> **⚠️ 注意**：MQ 消费者中的通知 SQL 不能按 `enrollment.status = 'ENROLLED'` 过滤，因为事务已将所有 enrollment 更新为 `APPROVED`。应按 `offering_id` 查询所有关联记录。
```

### 7.4 定时任务兜底

保留定时任务作为 Redis/MQ 不可用时的兜底方案：

```java
@Scheduled(fixedRate = 60_000)
public void autoUpdateCampaignStatus() {
    // PENDING → ACTIVE
    dbUpdateStatus(CampaignStatus.PENDING, CampaignStatus.ACTIVE);
    // ACTIVE → ENDED
    dbUpdateStatus(CampaignStatus.ACTIVE, CampaignStatus.ENDED);
    // 删除 Redis 缓存
    redisTemplate.delete("cas:campaign:current");
}
```

### 8.2 Token 黑名单（Redis 实现登出）

```java
// 登出时将 Token 加入 Redis 黑名单
public void logout(String token) {
    String jti = extractJti(token);  // JWT ID
    long ttl = getRemainingTtl(token);
    redisTemplate.opsForValue().set(
        "cas:token:blacklist:" + jti,
        "1",
        Duration.ofSeconds(ttl)
    );
}

// JWT 过滤器中检查黑名单
public void doFilterInternal(...) {
    String token = extractToken(request);
    String jti = extractJti(token);
    Boolean isBlacklisted = redisTemplate.hasKey("cas:token:blacklist:" + jti);
    if (Boolean.TRUE.equals(isBlacklisted)) {
        throw new BusinessException("Token 已失效");
    }
    // ... 正常校验
}
```

## 九、异常处理

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ApiResponse<Void> handleBusiness(BusinessException e) {
        return ApiResponse.error(409, e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ApiResponse<Void> handleValidation(MethodArgumentNotValidException e) {
        return ApiResponse.error(400, e.getBindingResult().getAllErrors().get(0).getDefaultMessage());
    }

    @ExceptionHandler(AccessDeniedException.class)
    public ApiResponse<Void> handleAccessDenied() {
        return ApiResponse.error(403, "无权限访问");
    }

    @ExceptionHandler(Exception.class)
    public ApiResponse<Void> handleUnknown(Exception e) {
        log.error("Unexpected error", e);
        return ApiResponse.error(500, "服务器内部错误");
    }
}
```
